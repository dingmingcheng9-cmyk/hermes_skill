# 跨会话连续性：单槽轮换摘要

## 为什么需要这个

Hermes Agent 的 `session_search` 工具搜索历史对话时有三个短板：

1. **FTS5 关键词匹配**，非语义搜索——用词不同就搜不到
2. **只返回 Top N 条**，跨天会话容易被埋没
3. **每次搜索额外消耗 token 和时间**，无法每个问题都搜

而 `memory` 工具虽然每次对话自动注入，但有 **2,200 字符上限**，堆太多历史条目会：
- 填满空间导致新信息存不进去
- 每条回复都带无关条目浪费 token

## 方案：单槽轮换摘要

只在记忆里保留 **1 条**「上次会话摘要」，固定位置，每次覆盖更新。

```
上次会话摘要: [日期] [主题] - [关键结论/状态]
```

### 对比分析

| | 完整记忆历史 | session_search | 单槽轮换摘要 |
|---|---|---|---|
| **每次 token 成本** | 每次均携带，随条目增长 | 仅搜索时产生 | 约 100 字 / 25-35 tokens，固定不变 |
| **跨日连续性** | ✅ 能存多条 | ❌ 需主动搜索、可能漏 | ✅ 看一眼就知道上次在干嘛 |
| **回溯深度** | ❌ 2,200 上限 | ✅ 不限量，搜到就有 | ❌ 只存上次，更早需 session_search |
| **维护成本** | ❌ 需定期整理 | ✅ 零维护 | ✅ 自动覆盖 |

### 规则

1. **何时更新**：每次完成「复杂任务」后覆盖更新（5+ 工具调用的任务、有结论的项目、用户明确说"下一个"时）
2. **不更新**：日常闲聊、一问一答的小问题、无结论的中间步骤
3. **格式**：`上次会话摘要: [日期] [主题] - [一句话结论/状态]`
4. **回溯更早记录**：仍通过 `session_search` 查询（关键词搜或让用户提示上下文）

### 执行步骤

```python
# 1. 完成复杂任务后
# 2. 用 memory(action='replace') 覆盖已有条目
memory(action='replace',
       target='memory',
       old_text='上次会话摘要',
       content='上次会话摘要: 2026-05-25 跨会话连续性讨论 - 用户采用单槽轮换摘要方案，已存档为技能')
```

### 配套使用方式

- **新对话开始时**：直接从记忆读到上次摘要，无需搜
- **需要回溯更早记录时**：用户说"之前那个 X 的事情" → 我调用 `session_search`
- **记忆空间足够时**：还能存 ~500 字符的环境事实（不冲突）

### 踩坑点

- ❌ 不要一个对话内频繁覆盖——只在有明确结论时更新
- ❌ 不要存详细步骤——摘要只需告知「之前聊了什么」，具体内容用 `session_search` 查
- ✅ 摘要里加日期，方便判断时效性
- ✅ 存"待办"或"中断"状态，方便下次接续
