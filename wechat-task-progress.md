# 微信端长任务进度透明化协议

用户在微信上等待超过 15 秒看不到进展 = 体验问题。执行复杂任务时必须让用户知道进度。

## 触发条件

当用户请求的任务满足以下任一条件时 **必须** 执行本协议：
- 3 个以上的主要步骤
- 可能耗时超过 30 秒
- 需要多次工具调用（下载、编译、等待服务等）

**不触发**的场景：简单问答（1-2步）、只读操作（查文件/搜东西）、用户说不用报进度。

## 协议流程

### ① 任务开始前 → 发计划

先发一条消息，格式：
```
📡 [任务名]：① 步骤一 → ② 步骤二 → ③ 步骤三 → ④ 步骤四（预计X分钟）
```

不要用长句描述，用户扫一眼就懂。

### ② 执行中 → 关键节点报进度

**消息格式（一行）：**
```
📡 [任务名]：① ✓ ② 下载中... → ③ 待执行 → ④ 待执行
```

**规则：**
- 只对**确实耗时**的步骤（>10秒）单独报进度
- 连续快速小步骤（<5秒/个）合并为一次更新
- 每完成一步，更新进度条消息
- **步骤失败**立即告知：
  ```
  ❌ [任务名]：② 下载失败（超时），正在重试第2次...
  ```
  或：
  ```
  ⚠️ [任务名]：③ 遇到问题已跳过，不影响后续
  ```
- **动态预计时间**：前几步走完后可修正剩余时间：
  ```
  📡 [任务名]：① ✓ ② ✓ → ③ 启动中（上步较快，剩余约30秒）
  ```

### ③ 任务完成 → 总结报告

```
✅ [任务名] 完成！耗时X分X秒

成果：
- [结果1]
- [结果2]

关键命令：
`xxxxx`

文件位置：/path/to/file
```

## 格式约定

- 所有进度消息统一用 **emoji + 任务名** 开头（📡 进行中 / ✅ 完成 / ❌ 失败 / ⚠️ 跳过）
- 步骤编号：① ② ③ ④ ...
- 完成状态：✓
- 进行中状态：...
- 每行控制在微信一屏可见，不要刷屏
- 进度消息用 `send_message` 工具发送——**非阻塞**，发完继续干活

## 示例

```
📡 配置代理：① 下载软件 → ② 写配置 → ③ 启动服务 → ④ 测试（约1-2分钟）
📡 配置代理：① ✓ ② ✓ → ③ 启动服务中... → ④ 待执行
📡 配置代理：① ✓ ② ✓ ③ ✓ → ④ 测试连通性...
✅ 配置代理完成！耗时1分15秒

状态：代理已运行，SOCKS5 :1080
命令：curl -x socks5://127.0.0.1:1080 google.com
配置：/opt/sing-box/config.json
```
